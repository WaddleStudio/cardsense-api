## [HIGH] 遷移至 Supabase (PostgreSQL)

**What:** 將後端資料庫從 SQLite 遷移至 Supabase。

**Why:** 使用者已確認將用 Supabase 取代 SQLite。SQLite 的並發限制（單寫入線程、file locking）在多請求場景下會造成鎖定失敗。Supabase 是 Railway 部署環境的合理選擇。

**How to apply:** 此項目未完成前，不要對 SQLite 連線管理進行深度重構。

**Subtasks:**
1. 在 `SqlitePromotionRepository` 旁建立 `SupabasePromotionRepository`（或重寫 SQL 語法為 PostgreSQL JSONB）
2. extractor 過程需要寫入 Supabase 而非本地 SQLite 檔案
3. 將 `dbPath` 設定屬性改為 Supabase JDBC URL / 連線字串
4. 定義 Spring profile：`local`=SQLite, `prod`=Supabase（允許本地開發繼續使用 SQLite）
5. 將 SQLite 特有語法（`json_extract`）遷移為 PostgreSQL JSONB 語法（`->>`）

**Depends on:** 需要同步更新 extractor repo。

## [HIGH] 確認 POINTS 型促銷的 rate 欄位語意

**What:** 釐清 DB 中 POINTS 型促銷的 `cashback_value` 欄位是「積點/元」還是「等效現金折扣 %」。

**Why:** `RewardCalculator` 目前以 `amount × rate / 100` 計算 POINTS，與 PERCENT 完全相同。若 rate 儲存的是「積點/元」（如 3 點/元），則計算語意錯誤——積點在台灣兌換率通常為 0.3–1%，且有最低兌換門檻與效期。若 rate 已換算為等效現金折扣 %，則目前計算正確，但應加上文件說明。

**How to apply:** 在 extractor repo 中確認 POINTS 型促銷的 `cashback_value` 定義，並在此 repo 加上 inline 註解說明。若需修正，同時更新 `RewardCalculator` 並補測試。

**Depends on:** 需查閱 extractor repo 的資料模型定義。

## [MEDIUM] API 限流

**What:** 為 `POST /v1/recommendations/card` 加入請求限流（rate limiting）。

**Why:** 目前 API 無任何限流。前端上線後，惡意或意外的高頻請求會無限消耗 Railway 資源（CPU + 出口流量費用）。

**How to apply:** 等到有實際流量資料後再決定限制閾值。建議使用 Spring `bucket4j` 或 Railway 層級的 throttle 配置。

**Depends on:** 有前端後再評估實際流量模式。

## [MEDIUM] Bitmask >15 促銷降級是排名正確性問題

**What:** 當一張卡有 >15 個符合條件的促銷時，`resolveContributingPromotions` 直接回傳排名第一的單一促銷，不做組合計算。

**Why:** 這不只是效能問題——擁有 16 個促銷的卡會得到比只有 14 個促銷的卡更低的預估回饋，即使前者實際上能疊加更多優惠。目前資料（每張卡每類別 2-4 個促銷）幾乎不觸發此路徑，但資料庫成長後會成為排名 bug。

**How to apply:** 修復方式：在 bitmask 搜尋前，先按 `cappedReturn` 排序取前 5 名促銷，只對這 5 個做組合計算。這保留了算法的準確性並移除了靜默降級。

**Depends on:** 資料庫成長到單卡單類別超過 10 個促銷時優先處理。
